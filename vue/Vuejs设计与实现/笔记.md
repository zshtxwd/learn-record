# Vuejs设计与实现

## 权衡的艺术

-----

### 命令式与声明式

*框架的两大范式，命令式和声明式*

#### 命令式框架

命令式是实现的过程，代码本身描述的是“做事的过程”

#### 声明式框架

声明式不注重过程，我们提供的是一个结果，框架根据这个结果去自动封装代码实现

------

### 性能与可维护性

*声明式的代码性能不优于命令式的代码性能*

如果我们把直接修改的性能消耗定义为 A，把找出差异的性能消耗定义为 B，那么有：

- 命令式代码的更新性能消耗 = A

- 声明式代码的更新性能消耗 = B + A

当找出差异的性能消耗无限趋近于0，那么声明式的代码性能等于命令式的代码性能，但是无法做到超越，**毕竟声明式框架本身就是内部封装了命令式代码才实现了面向用户的声明式**，因此声明式的代码性能不优于命令式的代码性能

**vue**采用了**声明式**，原因在于声明式的代码可维护性高于命令式的代码

在使用命令式的框架去实现功能的时候，我们要实现维护功能的整个过程，而声明式代码不关心过程，代码展示的就是我们想要的结果，我们不关心代码的实现，vue内部都帮我们封装好了

在采用声明式提升可维护性的同时，性能就会有一定的损失，而框架设计者要做的就是：**在保持可维护性的同时让性能损失最小化**

-----

### 虚拟DOM的性能

*声明式代码的更新性能消耗 = 找出差异的性能消耗 \+ 直接修改的性能消耗*

因此，如果我们能够最小化找出差异的性能消耗，就可以让声明式代码的性能无限接近命令式代码的性能。而所谓的**虚拟 DOM**，就是为了最小化找出差异这一步的性能消耗而出现的

采用**虚拟 DOM** 的更新技术的性能**理论上**不可能比**原生 JavaScript 操作 DOM** 更高。

因为在大部分情况下，我们很难写出绝对优化的命令式代码，尤其是当应用程序的规模很大的时候，即使你写出了极致优化的代码，也一定耗费了巨大的精力，这时的投入产出比其实并不高。

有没有什么办法能够让我们不用付出太多的努力（写声明式代码），还能够保证应用程序的性能下限，让应用程序的性能不至于太差，甚至想办法逼近命令式代码的性能呢？这其实就是虚拟 DOM要解决的问题。

#### 创建页面的性能

##### innerHTML创建页面的性能

在早年使用 **jQuery** 或者直接使用 JavaScript 编写页面的时候，使用 **innerHTML **来操作页面非常常见

使用 innerHTML 操作页面和虚拟 DOM 相比性能如何？innerHTML 和document.createElement 等 DOM 操作方法有何差异？

```javascript
// 对于 innerHTML 来说，为了创建页面，我们需要构造一段 HTML 字符串
const html = `
<div><span>...</span></div>
`
// 接着将该字符串赋值给 DOM 元素的 innerHTML 属性
div.innerHTML = html
```
为了渲染出页面，首先要把字符串解析成 DOM 树，这是一个 DOM 层面的计算，涉及DOM 的运算要远比 JavaScript 层面的计算性能差

纯**JavaScript** 层面的操作要比 **DOM **操作快得多，它们不在一个数量级上

*通过 innerHTML 创建页面的性能 = HTML 字符串拼接的计算量 + innerHTML 的 DOM计算量*

##### 虚拟DOM创建页面的性能

虚拟 DOM 创建页面的过程分为两步：第一步是创建 JavaScript 对象，这个对象可以理解为真实 DOM 的描述；第二步是**递归地遍历**虚拟 DOM 树并创建真实 DOM

我们同样可以用一个公式来表达：

*虚拟DOM创建页面的性能 = 创建 JavaScript 对象的计算量 + 创建真实 DOM 的计算量*

#####  创建页面时的性能对比

|                  |          虚拟DOM          |    innerHTML    |
| :--------------: | :-----------------------: | :-------------: |
| 纯JavaScript运算 | 创建JavaScript对象(Vnode) | 渲染HTML字符串  |
|     DOM运算      |      新建所有DOM元素      | 新建所有DOM元素 |

无论是纯 JavaScript 层面的计算，还是 DOM 层面的计算，其实两者差距不大

这里我们从宏观的角度只看数量级上的差异，如果在同一个数量级，则认为没有差异。

***innerHTML**和**虚拟 DOM**在创建页面的时候，都需要**新建所有 DOM 元素***

> 创建页面时的性能情况，大家可能会觉得虚拟DOM 相比 innerHTML 没有优势可言，甚至细究的话性能可能会更差。别着急，接下来我们看看它们在更新页面时的性能。

#### 更新页面的性能

##### innerHTML更新页面的性能

使用 innerHTML 更新页面的过程是**重新构建 HTML 字符串，再重新设置 DOM 元素的 innerHTML 属性**，哪怕我们只更改了一个文字，也要重新设置 innerHTML 属性。而重新设置innerHTML 属性就等价于**销毁所有旧的 DOM 元素，再全量创建新的 DOM 元素**

##### 虚拟DOM更新页面的性能

虚拟DOM更新页面时会**重新创建 JavaScript 对象（虚拟 DOM 树）**，然后**比较新旧虚拟 DOM**，找到**变化的元素**并**更新**它

##### 更新页面时的性能对比

在更新页面时，虚拟 DOM 在 JavaScript 层面的运算要比创建页面时多出一个 **Diff** 的性能消耗，然而它毕竟也是**JavaScript 层面**的运算，所以不会产生**数量级的差异**。

|                  |           虚拟DOM           |            innerHtml             |
| :--------------: | :-------------------------: | :------------------------------: |
| 纯JavaScript运算 | 创建新的JavaScript对象+Diff |                                  |
|     DOM运算      |        必要的DOM更新        | 销毁所有旧DOM<br />新建所有新DOM |
|     性能因素     |      与数据变化量相关       |          与模板大小相关          |



##### innerHTML、虚拟 DOM 以及原生 JavaScript在更新页面时的性能以及性能因素

​													  		**innerHTML 		  	   > 			    虚拟DOM		  	 > 			   	原生JavaScript**

|  | innerHTML | 虚拟DOM | 原生JavaScript |
| :-------:| :-------: | :-----: | :------------: |
| 性能 | 性能差 | 性能不错 |性能好|
| 可维护性 |         | 可维护性强 |可维护性差|
| 心智负担 | 心智负担中等 | 心智负担小 |心智负担大|

### 运行时和编译时

设计一个框架的时候，我们有三种选择：

- 纯运行时的
- 运行时 + 编译时的
- 纯编译时的

#### 纯运行时

- 用户输入的数据结构在 **浏览器运行时** 直接生成页面。  
- 框架核心是一个 `Render` 方法：接收用户输入的数据结构，输出页面内容。  
- 特点：灵活，但性能依赖运行时。[^灵活性]: 页面结构在运行时是否还能根据未知数据或逻辑动态生成/改变

#### 编译时+运行时

- 用户输入需要先经过 `Compiler` 方法，转化为命令式代码。  
- 分两种情况：
  - **运行时编译**：编译发生在浏览器中。
  - **编译时 + 运行时**：编译在构建阶段完成，浏览器仅执行编译后的命令式代码。
- 框架包含 `Compiler` 与 `Render` 方法：
  - 输入 → `Compiler` 编译成中间数据结构/命令式代码 → `Render` 生成页面。
  - 也可直接调用 `Render` 方法。
- 特点：性能和灵活性兼顾。

#### 纯编译时

- 用户输入在 **构建阶段** 就被编译为命令式代码。  
- 浏览器运行时无需再执行页面结构生成逻辑，只需执行少量框架运行时代码（如事件绑定、状态更新等）。  
- 框架核心仅包含 `Compiler` 方法。  
- 特点：运行时开销最小，但灵活性最低。

### 纯运行时和编译时+运行时的灵活性差异

#### 纯运行时

- 模板/数据结构在 **运行时（浏览器中）** 被直接解释执行。
- 框架会在浏览器里内置一个“解释器”（比如 JSX 解释、虚拟 DOM diff 引擎）。
- **好处**：模板可以非常动态，运行时才决定最终页面长啥样。
- **代价**：解释器必须一直在运行，性能开销大。
- **体现**： 
  - **运行时有模板解释器**，所以你可以在浏览器端根据任意数据结构「自由生成页面结构」。
  - **自由度**：不仅能改数据，还能改结构。
  - 例如：
    - 运行时接收一个新 JSON，拼一个新的 DOM 树。
    - 运行时决定标签是 `<ul>` 还是 `<table>`，甚至完全替换掉页面结构。
  - 框架代表：早期 React（JSX 本质是运行时解释）、一些动态 JSON → DOM 的低代码框架。

#### 编译时 + 运行时

- 模板在 **编译阶段** 就被转换成 **渲染函数（JavaScript 可执行代码）**。
- 浏览器只运行渲染函数，不需要在运行时再解释模板。
- 但运行时依然需要去执行渲染函数，根据数据更新 DOM。
- **好处**：运行时压力减轻（少了“解释模板”的步骤）。
- **限制**：模板的结构已经在编译时固定了，运行时只能在这个“框架”里更新数据。
- **模板在编译阶段就被固化为渲染函数**。
- 浏览器运行时只执行渲染函数 → 更新 DOM。
- **自由度**：只能在编译好的“结构框架”内更新数据，不能随便改结构。
- 例如：
  - 模板 `<div>{{ msg }}</div>` 编译成渲染函数后，运行时只能改 `msg` 的内容，不能再把 `<div>` 换成 `<ul>`，除非你在编译时就写好了条件分支。
- 框架代表：Vue（模板编译 → 渲染函数）、现代 React（JSX 编译为 `React.createElement` 调用）。

- 详细的警告信息方便开发者定位问题所在
- 友好的控制台信息输出，例如ref变量的formatter

### 控制框架代码的体积

框架的大小也是衡量框架的标准之一。在实现同样功能的情况下，当然是用的代码越少越好，这样体积就会越小，最后浏览器加载资源的时间也就越少。这时我们不禁会想，提供越完善的警告信息就意味着我们要编写更多的代码，这与控制代码体积相悖
Vue.js 使用 rollup.js 对项目进行构建，这里的_\_DEV\_\_ 常量实际上是通过 rollup.js 的插件配置来预定义的，其功能类似于 webpack 中的DefinePlugin 插件。

```javascript
if (__DEV__ && !res) {
 warn(
 	`Failed to mount app: mount target selector "${container}" 
returned null.`
 )
}
```

Vue.js 在输出资源的时候，会输出两个版本，其中一个用于开发环境，如 vue.global.js，另一个用于生产环境，如 vue.global.prod.js
当 Vue.js 构建用于开发环境的资源时，会把 _\_DEV\_\_ 常量设置为true
当 Vue.js 构建用于生产环境的资源时，会把 _\_DEV\_\_ 常量设置为false
```javascript
if(false){warn('xxx')}
```
**Dead Code Elimination（死代码消除）**：构建工具在打包的压缩/优化阶段会做静态分析，发现这种永远不会执行的分支会直接删除

- **Tree Shaking**：针对 **模块导入导出**，如果某个函数/变量没有被用到，就会在打包时删除。
- **Dead Code Elimination**：针对 **永远不会执行的代码分支**，直接移除。
两者都是 **减少生产包体积** 的优化手段，但逻辑不一样。

### 框架要做到良好的 Tree-Shaking

上文提到通过构建工具设置预定义的常量 _\_DEV\_\_，就能够在生产环境中使得框架不包含用于打印警告信息的代码，从而使得框架自身的代码量不随警告信息的增加而增加。但是从用户的角度来看，这么做仍然不够，Vue.js 内建了很多组件，例如 \<Transition\> 组件，如果我们的项目中根本就没有用到该组件，那么它的代码不需要包含在项目最终的构建资源中，那么可以通过Tree-Shaking做到这一点呢

#### Tree-Shaking的定义

在前端领域，Tree-Shaking这个概念因 rollup.js 而普及。简单地说，Tree-Shaking 指的就是消除那些永远不会被执行的代码，也就是排除 dead code，现在无论是 rollup.js 还是 webpack，都支持Tree-Shaking。

#### Tree-Shaking的实现

想要实现 Tree-Shaking，必须满足一个条件，即模块必须是ESM（ES Module），因为 Tree-Shaking 依赖 ESM 的静态结构。

- 必须是 **ESM（ES Module）** 格式，因为它的 `import/export` 是静态的，可以在打包阶段分析依赖关系。
- CommonJS 是动态的（`require` 可以写在函数里，路径也可能是变量），难以静态分析，所以 Tree-Shaking 不好做。

以 **rollup.js** 为例看看 Tree-Shaking 如何工作，其目录结构如下：

```javascript
├── demo
│    └── package.json
│    └── input.js
│    └── utils.js
```

首先安装 rollup.js：

```shell
 npm install rollup -D
```

下面是 input.js 和 utils.js 文件的内容：

```javascript
// input.js
import { foo } from './utils.js'
foo()
// utils.js
export function foo(obj) {
obj && obj.foo
}
export function bar(obj) {
obj && obj.bar
}
```

在 utils.js 文件中定义并导出了两个函数，分别是 foo 函数和 bar 函数，然后在 input.js 中导入了 foo 函数并执行。注意，并没有导入 bar 函数。

接着，执行如下命令进行构建：

```shell
 npx rollup input.js -f esm -o bundle.js
```

这句命令的意思是，以 input.js 文件为入口，输出 ESM，输出的文件叫作 bundle.js。命令执行成功后，打开 bundle.js 来查看一下它的内容：

```javascript
// bundle.js
function foo(obj) {
obj && obj.foo
}
foo()
```

可以看到，其中并不包含 bar 函数，这说明 Tree-Shaking 起了作用。由于并没有使用 bar 函数，因此它作为 dead code 被删除了。但是foo 函数的执行也没有什么意义，仅仅是读取了对象的值，所以它的执行似乎没什么必要。既然把这段代码删了也不会对应用程序产生影响，那么为什么 rollup.js 不把这段代码也作为 dead code 移除呢？

#### 副作用

如果一个函数调用会产生**副作用**，那么就不能将其移除。

##### 什么是副作用

简单地说，副作用就是，***当调用函数的时候会对外部产生影响***，例如修改了全局变量。

为什么 上文中`foo()` 的调用没有被删除？

- 因为 Tree-Shaking 不能确定这个调用是否会产生副作用。
- 举例：如果 `obj` 是 Proxy，那么 `obj.foo` 可能触发 `get` 拦截器，修改全局状态。
- **结论**：打包工具不敢轻易删除看似“无用”的代码，因为它可能影响程序行为。

#### 解决办法 `/*#__PURE__*/` 注释

因为静态地分析 JavaScript 代码很困难，所以像 rollup.js 这类工具都会提供一个机制，让开发者能明确地告诉 rollup.js：“放心吧，这段代码不会产生副作用，你可以移除它。”具体怎么做呢？如以下代码所示，修改 input.js 文件：

```javascript
import {foo} from './utils'
/*#__PURE__*/ foo()
```

注释代码 `/*#__PURE__*/`，其作用就是告诉 rollup.js，对于foo 函数的调用不会产生副作用，可以放心地对其进行 TreeShaking，此时再次执行构建命令并查看 bundle.js 文件，就会发现它的内容是空的，这说明 Tree-Shaking 生效了。

基于这个案例在编写框架的时候需要合理使用`/*#__PURE__*/` 注释，	Vue.js3的源码中就大量使用了该注释

```javascript
export const isHTMLTag = /*#__PURE__*/ makeMap(HTML_TAGS)
```

#### 顶级调用

手动添加该注释并不会对编写代码造成很大的心智负担，因为通常产生副作用的代码都是模块内函数的顶级调用。

##### 什么是顶级调用

**顶层调用** = 写在模块最外层的执行代码（***模块一加载就会运行***）。

**函数里的调用** ≠ 顶层调用（只有函数被用到才会运行）。

工具怕顶层调用有副作用 → 默认不删 → 需要 `/*#__PURE__*/` 手动声明无副作用。

```javascript
foo() // 顶级调用

function bar() {
foo() // 函数内调用
}
```

可以看到，对于顶级调用来说，是可能产生副作用的；但对于函数内调用来说，只要函数 bar 没有被调用，那么 foo 函数的调用自然不会产生副作用。

因此，在 Vue.js 3 的源码中，基本都是在一些顶级调用的函数上使用 `/*#__PURE__*/` 注释。当然，该注释不仅仅作用于函数，它可以应用于任何语句上。该注释也不是只有 rollup.js 才能识别，webpack 以及压缩工具（如 terser）都能识别它。

### 框架应该输出怎样的构建产物

Vue会为生产环境和开发环境输出不同的包，例如vue.global.js用于开发环境，包含必要的警告信息，而vue.global.prod.js用于生产环境，则没有这些警告信息，Vue.js的输出除了有环境上的区分，还有不同场景下的区分。

不同类型的产物一定有对应的需求背景

例如可以直接在HTML使用\<script\>标签直接引入并使用框架，为了实现这个需求，Vue需要输出一种叫做**IIFE**格式的资源

**IIFE** 的全称是 Immediately Invoked Function Expression，即“立即调用的

函数表达式”，即

```javascript
(function(){
    //...
}())
```

实际上，vue.global.js 文件就是 IIFE 形式的资源，它的代码结构如下所示：

```javascript
 var Vue = (function(exports){
 // ...
 exports.createApp = createApp
 // ...
 return exports
 }({}))
```

因此当我们使用\<script\>引入vue.global.js时，全局变量Vue就是可用的

在 rollup.js 中，我们可以通过配置 format: 'iife' 来输出这种形式的资源：

```javascript
 // rollup.config.js
 const config = {
 input: 'input.js',
 output: {
 file: 'output.js',
 format: 'iife' // 指定模块形式
 }
 }

 export default config
```
