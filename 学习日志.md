##### function* 生成器函数

这种声明方式 (function关键字后跟一个星号）会定义一个生成器函数 (generator function)，它返回一个 Generator 对象。

```javascript
function* generator(i) {
  yield i;
  yield i + 10;
  yield i + 100;
}

const gen = generator(1);

console.log(gen.next().value);
// Expected output: 1

console.log(gen.next().value);
// Expected output: 11

console.log(gen.next().value);
// Expected output: 101

console.log(gen.next().value);
//Expected output: undefined
```

##### ?.可选链操作符

?. 是可选链操作符（Optional Chaining Operator），它是 JavaScript 中的一个语法特性。它用于在访问对象属性时进行安全的、避免错误的链式操作。

当你使用 ?. 访问对象属性时，它会在属性链中进行安全导航。***如果遇到 null 或 undefined 的属性，表达式的结果将为 undefined，而不会引发错误***。这对于处理可能存在不存在的属性的情况非常有用。

在你提供的代码中，props.info?.status 表达式中的 ?. 用于访问 props.info 对象的 status 属性，如果 props.info 为 null 或 undefined，那么整个表达式的结果将为 undefined，而不会导致错误。

使用可选链操作符 ?. 可以简化代码，并避免在属性链中出现 null 或 undefined 时引发的错误。它在处理深层嵌套的对象属性时特别有用。

请注意，可选链操作符 ?. 是在 ECMAScript 2020（ES11）中引入的，因此确保你的 JavaScript 运行环境支持该特性。

```javascript
props.info?.status==1
```

这里就是如果info中没有status这个属性也不会报错，因为会将null当做undefined来进行处理

#### 2025的前段时间，具体忘记了



nextTick 的作用：等待 DOM 更新完成后再执行代码。因为 Vue 更新 DOM 是异步批处理的，修改响应式数据后 DOM 不会立刻更新。

为什么用：当你修改数据后，立刻想操作 DOM（例如获取高度、滚动、焦点、ref 等），如果不等 DOM 更新，会拿到旧数据或异常。

两种写法区别：

- nextTick(() => { ... })
   注册回调，等 DOM 更新后执行这段回调代码，不阻塞后续逻辑，DOM 更新后执行。

- await nextTick()
   原地等待 DOM 更新完成，然后继续执行下面的代码，会阻塞后面的逻辑直到更新结束，DOM 更新完再往下走。

核心理解：修改数据 → Vue 等待一轮事件循环 → 更新 DOM → 执行 nextTick



#### 2025/12/5



`ElTable` 提供了一个 `doLayout` 方法，用于重新计算表格的布局，包括：列宽、行高、滚动区域、固定列的位置

当表格结构或显示布局发生变化时（如显示/隐藏列、改变宽度），可以调用 `doLayout` 让表格重新渲染布局。

我的 `ElTable` 中有一列是文本内容，内部使用了 `textarea` 组件，并希望：`textarea` 的高度根据内容自动撑开

表格右侧还有一个“操作列”，通过编辑状态控制显示和隐藏。
 由于整个表格的宽度是固定的，显示和隐藏操作列时，内容列的宽度会变化：

| 状态       | 内容列宽度 | textarea 显示效果                                    |
| ---------- | ---------- | ---------------------------------------------------- |
| 没有操作列 | 宽         | 一行就能装下内容，高度只有一行                       |
| 显示操作列 | 变窄       | 文字换行变成两行，但 textarea 高度未更新，出现滚动条 |

在切换操作列显示状态后：

1. 先调用 `tableRef.doLayout()` 重新计算表格布局，让内容列宽度更新
2. 再调用 textarea 的 `resizeTextarea()` 方法，让其高度根据新宽度重新自适应

> 两步要依次执行，因为 `resizeTextarea` 必须在新宽度确定之后再触发，否则计算不到正确高度。

示例：

```javascript
nextTick(() => {
  tableRef.value?.doLayout()
  textareaRef.value?.resizeTextarea()
})
```

`doLayout()` 主要用于 **重新布局表格**，而不是直接解决文本域高度问题。
 真正让 textarea 重新适应内容高度的是 `resizeTextarea()`
 只是在 **列宽变化后必须先触发表格布局重算** 才能正确计算 textarea 的高度，所以两者结合使用。





#### 2026/1/12

##### 错误反省：使用ref包裹chart导致，echarts内部的属性或变量无法正常访问导致报错

在使用datazoom的inside做可拖动x轴功能时报错：`Uncaught (in promise) TypeError: chart.getCoordinateSystems is not a function`

dataZoom / 拖动滚动无法使用

图表初始化部分虽然能渲染，但交互异常

经过排查发现：

- 当 ECharts 实例被 `ref` 包裹时，它会变成 **响应式代理对象**
- ECharts 内部依赖原生实例的属性和方法（如 `_model`、`getCoordinateSystems` 等）
- 响应式代理会破坏这些内部访问，从而导致方法不存在或行为异常

##### 解决方案

1. **普通变量**

   - 最直接：不要用 `ref` 包裹 ECharts 实例

   ```javascript
   let chart = null
   chart = echarts.init(dom)
   chart.setOption(option)
   ```

   - 这样 ECharts 内部状态和方法不受代理影响，滚动和交互正常。

2. **shallowRef**（浅响应式）

   - 如果仍希望在模板或逻辑里保持引用响应式：

   ```javascript
   import { shallowRef } from 'vue'
   const chart = shallowRef(null)
   chart.value = echarts.init(dom)
   chart.value.setOption(option)
   ```

   - `shallowRef` 只代理 `.value`，**不代理内部属性**，避免破坏 ECharts 内部结构。

##### 总结教训

1. **区分数据类型**
   - Vue 响应式系统适合普通数据（数字、数组、对象）
   - 第三方库实例或复杂对象（ECharts、Mapbox、Three.js 等）不要深度响应式包装
2. **选择合适的响应式工具**
   - 不需要模板响应或深度监听 → 用普通变量
   - 需要引用响应式 → 用 `shallowRef`
3. **调试思路**
   - 报错看对象方法是否存在 → 很可能是响应式代理破坏了内部状态
   - 可以尝试将 `ref` 替换为普通变量或 `shallowRef` 测试

> ✅ 教训：**不要对 ECharts 实例用深度响应式 ref，使用普通变量或 shallowRef 才安全**
